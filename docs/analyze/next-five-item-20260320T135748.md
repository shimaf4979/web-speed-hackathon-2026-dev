## 4. 実装前の仮説

- 現状の `useInfiniteFetch` は client 側で全件取得してから `slice(offset, offset + LIMIT)` していた。
- ホームの `/api/v1/posts` は初回表示でも全件を返しており、Lighthouse 上では約 `2.56 MB` まで膨らんでいた。
- hook を generic に直すと検索件数表示や他ページの VRT まで変わるため、今回はホームだけ opt-in で server pagination を使う方が安全。

## 5. 採用した方針

- 安全優先案を採用した。
- `useInfiniteFetch` に `serverPagination` オプションを追加し、既存挙動はデフォルト維持。
- `TimelineContainer` だけ `serverPagination: true` を指定する。
- これにより:
  - ホームだけ `/api/v1/posts?limit=30&offset=...` に切り替える
  - 検索、ユーザー詳細、コメント一覧の表示仕様は変えない

## 6. 実装内容

- `application/client/src/hooks/use_infinite_fetch.ts`
  - `Options` を追加
  - `serverPagination` が有効なときだけ `limit` / `offset` を query に付与
  - server pagination 時は `slice` をやめ、返却ページをそのまま積み上げる
  - 最終ページ判定を `nextItems.length < LIMIT` で管理
  - `apiPath === ""` のときは fetch しない
- `application/client/src/containers/TimelineContainer.tsx`
  - `useInfiniteFetch(..., { serverPagination: true })` に変更
